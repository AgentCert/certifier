SECTION A

What Changes When SLAs Are Defined?

The No-SLA framework (H-01 through H-05) estimates performance and compares across categories without predefined thresholds. When SLAs are defined per fault type, we gain concrete pass/fail criteria, enabling stronger statistical claims and three entirely new hypothesis categories.

No-SLA Mode (H-01 to H-05)

  • Estimation-only: "What is the CI for TTD?"
  • Relative comparison: "Is category A different from B?"
  • No pass/fail threshold: Assessment is descriptive
  • 5 hypotheses: CI, Rate, Cross-Compare, Uniformity, Variance

SLA Mode (H-SLA-01 to H-SLA-08)

  • Threshold testing: "Does TTD meet the 300s SLA?"
  • Equivalence proving: "Is performance within SLA bounds?"
  • Formal pass/fail: Statistical evidence for compliance
  • 8 hypotheses: +Tail Risk, +Survival, +Drift, +Composite
Key Principle: SLA mode doesn't replace the No-SLA hypotheses — it extends and sharpens them. H-01's CI estimation becomes H-SLA-01's threshold comparison. H-02's rate floor becomes H-SLA-02's breach rate test. Three new hypotheses (Tail Risk, Survival Analysis, Drift Detection) address SLA-specific concerns that don't exist without thresholds.

SECTION B

The Eight SLA-Aware Hypotheses

SLA mode defines 8 hypotheses — 5 evolved from the No-SLA framework and 3 entirely new. Each hypothesis now has a concrete SLA threshold to test against.

Hypothesis Evolution Map
SLA Test From Question Method SLA Decision
H-SLA-01 H-01 Does typical performance meet SLA threshold? Bootstrap CI vs SLA + Wilcoxon signed-rank CI upper bound vs SLA threshold
H-SLA-02 H-02 Is the SLA breach rate acceptable? Exact binomial test + Wilson CI on breach rate Breach rate upper bound vs max allowed
H-SLA-03 H-03 Is performance provably within SLA bounds? TOST (Two One-Sided Tests) Both one-sided tests must reject
H-SLA-04 NEW How severe are SLA violations? CVaR (Conditional Value-at-Risk) Expected overshoot magnitude
H-SLA-05 H-04 Do all faults meet their individual SLAs? Per-fault Wilcoxon/TOST + BH-FDR Each fault vs its own SLA
H-SLA-06 NEW What is time-dependent SLA compliance? Kaplan-Meier survival + Log-rank S(SLA_threshold) ≥ target
H-SLA-07 H-05 Is SLA compliance stable or drifting? CUSUM + EWMA control charts No drift signal detected
H-SLA-08 NEW Overall SLA compliance composite Error budget + weighted composite All budgets within limits
H-SLA-01: SLA Threshold Compliance Test from H-01

Question: Does the agent's typical performance reliably meet the SLA threshold?

H₀: median(TTD) ≥ SLA_threshold (agent does NOT meet SLA)
Hₐ: median(TTD) < SLA_threshold (agent meets SLA)

Workflow:

  1. Compute IQM + Bootstrap BCa CI (same as H-01)
  2. Compare CI upper bound against SLA threshold
  3. Run one-sample Wilcoxon signed-rank test against SLA threshold
  4. Compute SLA margin: SLA_threshold - CI_upper_bound

Decision Rules:

PASS CI upper bound < SLA AND Wilcoxon p < 0.05
CONDITIONAL CI contains SLA threshold (uncertain)
FAIL CI lower bound > SLA threshold

Output Example:

App TTD: IQM=143s [128s, 159s] vs SLA=180s | Margin: +21s | Wilcoxon p=0.001 PASS
Net TTD: IQM=266s [127s, 418s] vs SLA=300s | Margin: -118s | Wilcoxon p=0.42 FAIL
Res TTD: IQM=258s [221s, 301s] vs SLA=600s | Margin: +299s | Wilcoxon p=0.001 PASS

Papers: Wilcoxon (1945), Efron (1979), Agarwal et al. (2021)

H-SLA-02: SLA Breach Rate Test from H-02

Question: Is the proportion of SLA-violating runs within acceptable limits?

H₀: breach_rate ≥ max_allowed_breach_rate
Hₐ: breach_rate < max_allowed_breach_rate

Workflow:

  1. Count runs where metric exceeds SLA threshold
  2. Exact binomial test: is breach rate < max allowed?
  3. Clopper-Pearson exact CI on breach rate
  4. Wilson CI on compliance rate (1 - breach_rate)
  5. Report minimum n needed to certify at given breach rate

Decision Rules:

PASS Breach CI upper bound < max_allowed AND binomial p < 0.05
CONDITIONAL Breach CI contains max_allowed threshold
FAIL Breach CI lower bound > max_allowed

Output Example:

App TTD breaches: 2/30 (6.7%) vs SLA max 5% | Clopper-Pearson CI: [0.8%, 22.1%] CONDITIONAL
Net TTD breaches: 12/30 (40.0%) vs SLA max 10% | Clopper-Pearson CI: [22.5%, 59.4%] FAIL
Res TTD breaches: 1/30 (3.3%) vs SLA max 10% | Clopper-Pearson CI: [0.1%, 17.2%] PASS

Papers: Clopper & Pearson (1934), Wilson (1927), Wald (1945)

H-SLA-03: TOST Equivalence Testing ENHANCED

Question: Can we prove the agent's performance is within SLA bounds (not just "not different")?

Key Distinction: Standard tests prove "different from SLA." TOST proves the opposite: "demonstrably within SLA bounds." This is the gold standard for certification claims.

H₀: Performance is OUTSIDE equivalence bounds [0, SLA_threshold]
Hₐ: Performance is WITHIN equivalence bounds [0, SLA_threshold]

Workflow:

  1. Define equivalence margin: [0, SLA_threshold]
  2. Test 1: H₀: μ ≤ 0 vs Hₐ: μ > 0 (lower bound check)
  3. Test 2: H₀: μ ≥ SLA vs Hₐ: μ < SLA (upper bound check)
  4. Both must reject at α=0.05 for equivalence to be declared

Decision Rules:

PROVEN WITHIN Both one-sided tests reject (p1 < 0.05 AND p2 < 0.05)
INCONCLUSIVE Only one test rejects
OUTSIDE BOUNDS Neither test rejects

Output Example:

App TTD vs [0, 180s]: TOST p1=0.001, p2=0.008 | max(p1,p2)=0.008 PROVEN WITHIN
Net TTD vs [0, 300s]: TOST p1=0.001, p2=0.312 | max(p1,p2)=0.312 INCONCLUSIVE
Res TTD vs [0, 600s]: TOST p1=0.001, p2=0.001 | max(p1,p2)=0.001 PROVEN WITHIN

Papers: Schuirmann (1987), Lakens (2017), Munk & Czado (1998)

H-SLA-04: SLA Violation Severity (Tail Risk) NEW

Question: When SLA is breached, how bad is the violation?

Why new: No-SLA mode has no threshold, so "violation severity" is undefined. With SLA, we can measure: "When TTD exceeds 300s, the average overshoot is 150s" — critical for incident response planning.

Metrics:

  • CVaR₅⁵ (Conditional Value-at-Risk): Average of the worst 5% of outcomes
  • Expected overshoot: E[X - SLA | X > SLA] — average amount by which violations exceed threshold
  • Max violation: Worst observed SLA breach
  • Violation ratio: CVaR / SLA_threshold (how many times worse than SLA)

Decision Rules:

MINOR Expected overshoot < 20% of SLA threshold
MODERATE Expected overshoot 20-100% of SLA threshold
SEVERE Expected overshoot > 100% of SLA threshold (violations are 2x+ the limit)

Output Example:

Net TTD (SLA=300s): CVaR95=891s | E[overshoot]=545s (182% of SLA) | Max=944s SEVERE
App TTD (SLA=180s): CVaR95=287s | E[overshoot]=107s (59% of SLA) | Max=287s MODERATE
Res TTD (SLA=600s): CVaR95=539s | No violations (0% breach) | Max=539s < SLA N/A

Papers: Rockafellar & Uryasev (2000), Artzner et al. (1999)

H-SLA-05: Per-Fault Individual SLA Compliance from H-04

Question: Does each fault type individually meet its own specific SLA?

Evolution from H-04: H-04 tested uniformity across categories (are rates equal?). H-SLA-05 tests each category against its own SLA (does each pass its own bar?). Different faults can have different SLAs.

Workflow:

  1. For each fault type, run H-SLA-01 against its specific SLA threshold
  2. Apply BH-FDR correction for multiple testing (k fault types)
  3. Identify the binding constraint (fault type closest to SLA violation)
  4. Compute per-fault error budget consumption

Output Example:

container-kill: TTD IQM=98s vs SLA=180s | Margin: +82s PASS | Budget: 46% used
pod-delete: TTD IQM=178s vs SLA=180s | Margin: +2s TIGHT | Budget: 99% used
pod-network-loss: TTD IQM=312s vs SLA=300s | Margin: -12s FAIL | Budget: EXCEEDED
disk-fill: TTD IQM=245s vs SLA=600s | Margin: +355s PASS | Budget: 41% used

Binding constraint: pod-network-loss (SLA exceeded by 12s) BH-FDR correction applied across 10 fault types

Papers: Benjamini & Hochberg (1995), Cochran (1977), Google SRE Book (2016)

H-SLA-06: Survival Analysis at SLA Threshold NEW

Question: What fraction of runs complete detection/mitigation before the SLA deadline?

Why new: Survival analysis properly handles censored data — runs where detection never occurred. Simple breach counting treats these as "infinite TTD," but survival analysis models them correctly as right-censored observations.

Workflow:

  1. Construct Kaplan-Meier survival curve S(t) per fault category
  2. Evaluate S(SLA_threshold) = P(TTD > SLA) = breach probability
  3. Compare curves across categories with Log-rank test
  4. Compute median survival time (time at which 50% of runs complete)

Output Example:

App: S(180s)=0.07 | 93% detect before SLA deadline | Median survival: 141s PASS
Net: S(300s)=0.60 | Only 40% detect before SLA deadline | Median survival: ∞ (censored) FAIL
Res: S(600s)=0.03 | 97% detect before SLA deadline | Median survival: 252s PASS
Log-rank test: χ²=24.3, p<0.001 | Survival curves differ significantly

Papers: Kaplan & Meier (1958), Mantel (1966)

H-SLA-07: SLA Compliance Drift Detection from H-05

Question: Is the agent's SLA compliance stable over time, or is it drifting toward violation?

Evolution from H-05: H-05 measured variance stability (CV). H-SLA-07 tracks directional drift toward SLA violation using control chart methods.

Workflow:

  1. Compute SLA margin per run: margin_i = SLA_threshold - observed_value_i
  2. Apply CUSUM chart on margin sequence (track cumulative drift)
  3. Apply EWMA smoothing (detect gradual degradation)
  4. Flag if CUSUM exceeds decision boundary or EWMA trend crosses zero
  5. Also report CV (retained from H-05) for variance stability

Decision Rules:

STABLE CUSUM within bounds AND CV < 0.30 AND EWMA margin > 0
DRIFTING CUSUM approaching boundary OR EWMA trend decreasing
ALERT CUSUM exceeded boundary OR EWMA margin < 0

Papers: Page (1954), Roberts (1959), Peters et al. (2016)

H-SLA-08: Composite SLA Score & Error Budget NEW

Question: Across all SLA dimensions and fault types, what is the overall compliance posture?

Why new: Individual hypothesis results need to be aggregated into a final certification decision. Error budgets (from Google SRE) provide a natural framework for tracking SLA consumption.

Components:

  • Error budget consumption: (observed_breach_rate / allowed_breach_rate) × 100%
  • Weighted composite: Per-SLA scores weighted by fault severity × frequency
  • Binding constraint: The single SLA dimension closest to (or past) violation
  • Overall verdict: ALL dimensions must pass for full certification

Output Example:

Error Budget Consumption:
  App detection_rate: 11% consumed (90% vs SLA 90%) OK
  Net detection_rate: 400% consumed (50% vs SLA 80%) EXCEEDED
  Res detection_rate: 67% consumed (70% vs SLA 75%) WARNING

Weighted Composite Score: 0.62 / 1.00 (weights: App=0.3, Net=0.4, Res=0.3)
Binding Constraint: Network detection_rate (50% vs SLA 80%)
Overall Verdict: CONDITIONAL — Network excluded from certification

Papers: Beyer et al. (2016) Google SRE, Benjamini & Hochberg (1995)

SECTION D

What Changed: Removed, Modified, and Added Hypotheses

Modified (5 Hypotheses)
  • H-01 → H-SLA-01: CI estimation gains SLA threshold comparison + Wilcoxon one-sample test
  • H-02 → H-SLA-02: Rate estimation gains breach rate testing via exact binomial
  • H-03 → H-SLA-03: Cross-category comparison replaced by TOST equivalence proving
  • H-04 → H-SLA-05: Uniformity test replaced by per-fault individual SLA testing
  • H-05 → H-SLA-07: Variance stability extended with CUSUM/EWMA drift detection
Added (3 New Hypotheses)
  • H-SLA-04 (Tail Risk): CVaR and expected overshoot — measures violation severity
  • H-SLA-06 (Survival): Kaplan-Meier analysis — handles censored data, time-dependent compliance
  • H-SLA-08 (Composite): Error budget aggregation — overall SLA posture across all dimensions
No-SLA Tests Not Needed in SLA Mode
  • Kruskal-Wallis (H-03): Replaced by per-SLA testing. "Are categories different?" is less relevant when each has its own threshold
  • Fisher's Exact (H-04): Uniformity across categories is less relevant when SLAs are heterogeneous
  • These tests can still be run as supplementary analysis but are not part of the core SLA verdict
Design Decision: The cross-category comparison tests (Kruskal-Wallis, Fisher's Exact) are demoted to supplementary in SLA mode because the question changes. Without SLA: "Is network worse than application?" (relative). With SLA: "Does network meet its own SLA?" (absolute). An agent can have dramatically different TTD across categories and still pass — if each meets its own SLA. The relative comparison becomes informational rather than gating.
