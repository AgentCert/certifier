One-Sample Wilcoxon Signed-Rank Test H-SLA-01

SLA threshold test · Non-parametric one-sample test against a fixed value
What Problem Does This Solve?
You have 30 detection times and an SLA that says "time_to_detect must be ≤ 300s." Is the agent reliably below 300s, or did it just get lucky this time? The one-sample Wilcoxon test formally tests whether the median of your data is significantly below the SLA threshold. Unlike a t-test, it doesn't assume bell-curve data — critical because agent response times are typically right-skewed with heavy tails.
How It Works
  1. Subtract the SLA threshold from each observation: di = xi - SLA
  2. If the agent meets SLA, most di values should be negative (below threshold)
  3. Rank the absolute values |di|, then assign + or - signs based on direction
  4. Sum the positive ranks (W+) and negative ranks (W-). Under H₀, they should be balanced
  5. If W- dominates (most values are below SLA), p-value is small → reject H₀
H₀: median(TTD) ≥ SLA_threshold (agent does NOT meet SLA) Hₐ: median(TTD) < SLA_threshold (agent meets SLA) Decision: If p < 0.05, reject H₀ — evidence that the agent reliably performs within SLA.
In Simple Terms
Imagine a pizza delivery guarantee: "30 minutes or it's free." You order pizza 30 times and record how long each delivery takes. The Wilcoxon test asks: "Is the median delivery time genuinely below 30 minutes, or did a few fast deliveries mask a generally slow service?" It looks at how far each delivery was from the 30-minute mark and weighs fast deliveries against slow ones. If most deliveries are comfortably under 30 minutes, the test says "yes, this pizza place reliably meets its SLA."

IssueOne-Sample t-TestOne-Sample Wilcoxon

Exact Binomial Test H-SLA-02

SLA breach rate test · Exact probability for binary compliance
When Every Violation Counts
Your SLA says "no more than 10% of runs should violate the TTD threshold." You observe 3 violations out of 30 runs (10%). Is this within budget, or is the true violation rate actually higher? The exact binomial test computes the exact probability of observing this many violations (or fewer) if the true rate equals the SLA limit. The Clopper-Pearson CI gives you the worst-case upper bound on the breach rate.
How It Works
  1. Count violations: x = number of runs where metric exceeds SLA threshold
  2. Total trials: n = total runs
  3. Test H₀: breach_rate ≥ max_allowed vs Hₐ: breach_rate < max_allowed
  4. Compute exact p-value using binomial distribution: P(X ≤ x | p = max_allowed)
  5. Clopper-Pearson CI: exact 95% bounds on the true breach rate
Exact Binomial Test: p-value = P(X ≤ x | n, p = max_allowed_breach_rate) If p < 0.05: evidence that breach rate is within SLA budget Clopper-Pearson 95% CI: [B(α/2; x, n-x+1), B(1-α/2; x+1, n-x)] where B is the Beta distribution quantile function
In Simple Terms
Imagine a factory quality control. The contract says "no more than 2% of widgets can be defective." You inspect 100 widgets and find 1 defective one (1%). But does that prove the defect rate is below 2%? The exact binomial test computes: "If the true defect rate were exactly 2%, what's the chance of seeing 1 or fewer defects in 100 inspections?" If that probability is high (not surprising), you can't be confident the rate is below 2%. The Clopper-Pearson CI tells you: "The true defect rate is between 0.03% and 5.4%" — the upper bound exceeding 2% means you can't certify at n=100.

This table explains why tight SLAs (≤5% breach) require either larger sample sizes or accepting wider CIs. At n=30 with 0 breaches, the Clopper-Pearson upper bound is still 11.6%.

TOST — Two One-Sided Tests H-SLA-03 NEW

Equivalence testing · Proving performance is WITHIN bounds
The Gold Standard for SLA Certification
Standard hypothesis tests can only prove "the agent is different from the SLA." But for certification, you want to prove the opposite: "the agent is within the SLA bounds." This reversal is called equivalence testing, and TOST is the FDA-approved method for it (used in drug bioequivalence trials). In our context: instead of failing to disprove compliance, we actively prove it.
How It Works
  1. Define the equivalence bounds: [0, SLA_threshold] for "less is better" metrics like TTD
  2. Test 1 (Lower bound): H₀: μ ≤ 0 vs Hₐ: μ > 0 — "Is performance above zero?" (trivially passes for times)
  3. Test 2 (Upper bound): H₀: μ ≥ SLA vs Hₐ: μ < SLA — "Is performance below the SLA threshold?"
  4. Both must reject at α=0.05 for equivalence to be declared
  5. The TOST p-value = max(p1, p2) — both tests must pass
TOST Decision: p_max = max(p_lower, p_upper) p_max < 0.05 → PROVEN WITHIN SLA bounds (strong claim) p_max ≥ 0.05 → INCONCLUSIVE (cannot prove compliance) Key: "Inconclusive" ≠ "Failing" — it means we lack statistical power to make the claim, possibly due to small n or high variance.
In Simple Terms
Imagine proving a drug is equivalent to an existing medication. The FDA doesn't accept "we failed to find a difference" — that could just mean the study was too small. Instead, TOST requires you to actively prove the drug's effect is within an acceptable range. Two guards check both ends: Guard 1 confirms "it's at least this effective" and Guard 2 confirms "it's not more than this effective." Both guards must give the OK. For SLA: Guard 1 says "the agent does respond" (trivial) and Guard 2 says "the response is within the SLA deadline" (the real test).
Why This Is Stronger Than CI Comparison
ApproachClaimStrength
CI upper < SLA"CI doesn't cross SLA"Informal — no formal p-value for the claim
Wilcoxon p < 0.05"Median is below SLA"Tests median, not mean/IQM; one-sided only
TOST p_max < 0.05"Performance is provably within [0, SLA]"Formal equivalence proof with power analysis

TOST also enables power analysis: given the observed variance, how many runs do we need to prove equivalence? This drives sample size recommendations.

Conditional Value-at-Risk (CVaR) H-SLA-04 NEW

Expected Shortfall · Tail risk measurement · "How bad is bad?"
When SLA Violations Happen, How Bad Are They?
Meeting SLA 90% of the time sounds good, but if the 10% violations are catastrophic (30-minute detection delays), that's a different story than if they're minor (32 seconds over the 300s limit). CVaR measures the average severity of the worst outcomes. It answers: "When things go wrong, how wrong do they go?" This is critical for incident response planning and SLA penalty estimation.
How It Works
  1. Identify the Value-at-Risk (VaR) at level α — the α-th percentile of outcomes (e.g., P95 = the 95th percentile)
  2. CVaR = average of all values worse than VaR. For TTD: average of the slowest 5% of detections
  3. Expected overshoot = E[X - SLA | X > SLA] — when SLA is violated, by how much on average?
  4. Overshoot ratio = expected overshoot / SLA threshold — how many times worse than the limit?
CVaR₀.₅⁵ = E[X | X > P95] = average of the worst 5% Expected Overshoot = E[X - SLA | X > SLA] = mean of (violation amount) for all violations Overshoot Ratio = Expected Overshoot / SLA_threshold = how many "SLA units" the typical violation is
In Simple Terms
Think of car insurance claims. The insurance company knows 5% of drivers will have an accident. But they need to know: when accidents happen, how expensive are they? CVaR answers this. If the average accident costs $5,000, that's manageable. If the average accident costs $500,000, the company needs very different reserves. For SLA: knowing that 10% of detections violate the 300s SLA is one thing. Knowing that those violations average 891s (nearly 3x the SLA) is a completely different risk profile.

Kaplan-Meier Survival Analysis H-SLA-06 NEW

Time-to-event analysis · SLA compliance over time · Handles censored data
The Clock Is Ticking: How Many Runs Beat the SLA Deadline?
Kaplan-Meier answers: "At any given time t, what fraction of runs have NOT yet detected the fault?" This creates a "survival curve" — the probability of still waiting at time t. At the SLA threshold time, this curve gives you the breach probability directly. Crucially, it properly handles censored data — runs where the fault was never detected (the agent gave up or timed out). Simple breach counting treats these as "infinite TTD," but Kaplan-Meier models them correctly.
How It Works
  1. List all event times (TTD values) and mark each as event (detected) or censored (never detected, timed out)
  2. At each event time ti, compute S(t) = S(t-1) × (1 - di/ni) where di = detections at ti, ni = runs still "at risk"
  3. The survival curve S(t) starts at 1.0 (all runs undetected) and decreases as detections occur
  4. At SLA threshold: S(SLA) = fraction of runs STILL undetected = breach probability
  5. Compliance at SLA = 1 - S(SLA) = fraction that detected within SLA deadline
At the SLA threshold: S(SLA_threshold) = P(TTD > SLA) = breach probability 1 - S(SLA_threshold) = P(TTD ≤ SLA) = compliance rate Log-rank test: Compares survival curves across fault categories H₀: Survival functions are identical across categories Hₐ: At least one category has a different survival pattern
In Simple Terms
Imagine tracking how long customers wait in a phone queue. Some hang up before being answered (censored — you know they waited at least X minutes, but not how long they would have waited). Kaplan-Meier handles this gracefully: it doesn't assume hang-ups would have been answered in 1 more minute or 1 hour — it honestly says "we don't know" and adjusts the curve accordingly. For fault detection: the 15 out of 30 network runs where the agent never detected the fault are "hang-ups" — Kaplan-Meier includes them in the survival estimate without pretending they would have detected at some specific time.

CUSUM Control Chart H-SLA-07 NEW

Cumulative Sum · Sequential drift detection · SLA margin monitoring
Catching SLA Drift Before It Becomes a Violation
An agent might meet SLA today but be slowly getting worse. CUSUM tracks the cumulative deviation from the SLA target — if the agent is consistently a few seconds slower than expected, the cumulative sum grows until it crosses a threshold, triggering an alert. It's like tracking your bank balance: individual small overdrafts might not alarm you, but the cumulative trend downward signals a problem.
How It Works
  1. For each run i, compute the SLA margin: mi = SLA_threshold - observed_valuei
  2. Positive margin = within SLA, negative margin = violation
  3. CUSUM: St = max(0, St-1 + (xt - k)) where k = acceptable slack
  4. If St exceeds decision boundary h, signal a drift alarm
  5. Also track EWMA (Exponentially Weighted Moving Average) for smoother trend detection
CUSUM (upper): St+ = max(0, St-1+ + (xt - μ₀ - k)) Signal: St+ > h → upward shift detected (getting slower) EWMA: Zt = λ · xt + (1 - λ) · Zt-1 λ = 0.2 (smoothing factor), Z₀ = SLA_threshold Signal: Zt approaches or crosses SLA threshold Combined decision: CUSUM in bounds AND EWMA margin > 0 → STABLE CUSUM approaching OR EWMA declining → DRIFTING CUSUM exceeded OR EWMA past SLA → ALERT
In Simple Terms
Think of a river level monitor near a dam. The water level might be fine right now, but CUSUM tracks the cumulative trend. If each day the river is 1 cm higher than expected, by day 30 the cumulative excess is 30 cm — even though no single day was alarming. CUSUM catches this creeping rise before the river overflows. For SLA: if each successive run is 5 seconds slower than the SLA midpoint, CUSUM detects the trend long before individual runs actually violate the SLA threshold. This gives you time to retrain or reconfigure the agent before customers are affected.
