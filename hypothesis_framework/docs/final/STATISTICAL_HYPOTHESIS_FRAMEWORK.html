Statistical Hypothesis Framework

Advanced Inference Strategy for AgentCert Certification Pipeline
SECTION A

What Is a Statistical Hypothesis Framework in AgentCert?

A statistical hypothesis framework replaces single-number summaries (means, rates) with formal probabilistic inference. Instead of asking "what is the mean time_to_detect?", we ask "what is the confidence interval around mean time_to_detect, does it meet SLA thresholds, how consistent is it across fault categories, and is there evidence of drift over time?"

Framework Context: This evaluates one agent using 30+ runs per fault category to ensure statistical validity. The framework operates in two modes:
  • No-SLA Mode: Provides confidence intervals, cross-category comparisons, effect sizes, and variance stability checks.
  • SLA-Aware Mode: Adds threshold compliance testing, breach rate estimation, equivalence testing, tail risk analysis, survival analysis, and drift detection against predefined SLA targets.

All 9 hypotheses always run. H-01 through H-05 provide the core evaluation. H-06 through H-09 activate additional SLA-specific tests when SLA thresholds are provided, and provide informational analysis even without SLAs.

What This Framework Delivers

Uncertainty Quantification

Wilson CI, Bootstrap CI, and IQM provide confidence intervals — not just point estimates.

Cross-Category Analysis

Kruskal-Wallis, Mann-Whitney U, Welch's ANOVA, and Fisher's Exact tests compare fault types.

Effect Size Measurement

Vargha-Delaney A₁₂ quantifies how large differences actually are — beyond statistical significance.

Stability Testing

Levene's test and CV analysis ensure consistent performance across runs.

SLA Threshold Compliance

Wilcoxon one-sample test and Bootstrap CI vs SLA threshold prove performance meets requirements.

SLA Tail Risk & Drift

CVaR quantifies worst-case severity. CUSUM/EWMA detect performance drift toward SLA violation.

SECTION C

The Nine Hypotheses: What We Test and Why

C.1 Understanding the Problem: Testing AI Agents Under Chaos

AgentCert evaluates how well AI agents handle real Kubernetes production failures by injecting controlled faults and measuring the agent's ability to detect, diagnose, and remediate them. We test across three fault categories:

Application Faults
  • container-kill — Pod crashes, CrashLoopBackOff
  • pod-delete — Pod termination, replica set reconciliation
Network Faults
  • pod-network-loss — Packet drop, partitions
  • pod-network-corruption — Data integrity failures
  • pod-network-rate-limit — Bandwidth throttling
  • pod-dns-error — DNS resolution failures
Resource Faults
  • disk-fill — Disk exhaustion, I/O pressure
  • pod-cpu-hog — CPU saturation, throttling
  • pod-memory-hog — Memory pressure, OOMKill
  • pod-autoscaler — HPA/VPA misconfiguration

C.3 Core Hypotheses (H-01 to H-05) — Always Active

H-01: Confidence Intervals for Continuous Metrics

“How fast does this agent actually respond — and how confident are we in that number?”

H₀: True typical performance is unknown — observed IQM is a noisy estimate.
Hₐ: Bootstrap CI provides a reliable bound on where the true performance lies.

Metrics: time_to_detect, time_to_mitigate, reasoning_score, hallucination_score

Workflow:

  1. Collect n=30-40 runs per fault category
  2. Compute: IQM (25% trimmed mean), Median, P95, Mean (reference)
  3. Bootstrap CI: Resample 10k times → distribution of IQMs → 95% CI

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

Benefit: IQM is robust to LLM retry loops (Agarwal NeurIPS 2021). Bootstrap doesn't assume normality.

time_to_detect: IQM=245s [210-280s BCa CI], median=242s, P95=452s, mean=293s

H-02: Success Rate Estimation with Safety Floor

“What percentage of faults does this agent actually catch — and what’s the worst-case guarantee?”

H₀: Observed success rate may overstate the true rate due to small sample size.
Hₐ: Wilson CI lower bound provides a conservative floor on the true success rate.

Metrics: fault_detection_success_rate, fault_mitigation_success_rate, false_negative_rate

Workflow:

  1. Count successes out of n=30-40 runs
  2. Wilson score interval → 95% CI
  3. Report lower bound as safety floor

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

Benefit: Wilson handles small n and extreme proportions (0%, 100%). Lower bound = conservative guarantee.

fault_detection_success_rate: 24/30 = 80.0% [62.1%-91.4% Wilson CI] | Floor: 62%

H-03: Cross-Category Performance Comparison

“Does this agent handle network faults just as well as application faults — or does it struggle with specific types?”

H₀: Performance is the same across all fault categories.
Hₐ: At least one category has significantly different performance.

Workflow:

  1. Kruskal-Wallis (omnibus): "Any difference?" → one p-value
  2. IF p<0.05: Mann-Whitney U pairwise + Holm-Bonferroni correction
  3. For each pair: Vargha-Delaney A₁₂ → effect size

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

KW H=18.4, p=0.0001 | Res vs App: MWU p=0.007, A₁₂=0.92 (large)

H-04: Cross-Category Success Rate Uniformity

“Does this agent fail network faults more often than application faults — or is the success rate roughly the same everywhere?”

H₀: Success rates are the same across all fault categories.
Hₐ: At least one category has a significantly different success rate.

Workflow:

  1. Build 2×3 contingency table
  2. Fisher's Exact Test → p-value
  3. If p<0.05: identify weakest category

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

App: 27/30 (90%) | Net: 15/30 (50%) | Res: 21/30 (70%) | Fisher p=0.003

H-05: Consistency & Predictability (Variance Stability)

“Can you rely on this agent to behave the same way every time — or is it fast one minute and slow the next?”

H₀: Variance is equal across all fault categories.
Hₐ: At least one category has significantly higher variance (erratic behavior).

Workflow: Compute CV per category. Levene's test for variance equality. CV: <0.15 stable | 0.15-0.30 moderate | >0.30 unstable

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

Net: σ=404s, CV=1.40 | Res: σ=106s, CV=0.38 | App: σ=52s, CV=0.14 | Levene p=0.02

C.4 SLA-Aware Hypotheses (H-06 to H-09)

Mode Note: H-06 and H-07 activate only when SLA thresholds are defined. H-08 and H-09 provide informational analysis even without SLAs.
SLA H-06: SLA Threshold Compliance

“Can we prove this agent meets the SLA — not just on average, but with statistical confidence?”

H₀: True median performance does NOT meet SLA (median ≥ threshold).
Hₐ: True median performance IS within SLA (median < threshold).

Workflow:

  1. Compute IQM + Bootstrap CI for each metric
  2. Compare CI upper bound against SLA threshold
  3. Wilcoxon one-sample test: H₀: median ≥ SLA vs Hₐ: median < SLA
  4. TOST equivalence: prove performance is within SLA bounds

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

PASS

CI upper ≤ SLA

CONDITIONAL

CI contains SLA

FAIL

CI lower > SLA

TTD (SLA ≤300s): IQM=245s [210-280s] | CI upper 280s < SLA 300s | Wilcoxon p=0.003 | PASS

SLA H-07: SLA Breach Rate Estimation

“Even if most runs are fine, do too many runs violate the SLA? Can we prove the breach rate is below the allowed limit?”

H₀: Breach rate ≥ allowed target (e.g., ≥ 5%).
Hₐ: Breach rate < allowed target.

Workflow:

  1. Count SLA breaches: runs where metric exceeds threshold
  2. Exact binomial test: H₀: breach_rate ≥ target vs Hₐ: breach_rate < target
  3. Wilson CI on breach rate
  4. Report minimum n needed to certify

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

Key Insight: At n=30, proving breach_rate ≤ 5% requires 0 observed breaches.

Breaches (TTD>300s): 3/30=10% [3.5%-25.6% CI] | Binomial p=0.184 | FAIL | Need n=93

H-08: Tail Risk Analysis BOTH*

“When this agent fails, how badly does it fail? Is a bad run just a little slow, or catastrophically slow?”

H₀: Tail outcomes are not disproportionately severe.
Hₐ: Worst 5% of runs are significantly worse than P95, indicating hidden catastrophic risk.

Workflow:

  1. CVaR₉₅: Average of worst 5% of outcomes
  2. P95 and P99 percentile values
  3. With SLA: Expected overshoot = E[X - SLA | X > SLA]
  4. Bootstrap CI on CVaR for uncertainty

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

Why: P95 is a single point; CVaR reveals "when things go wrong, how wrong?"

P95=452s | P99=890s | CVaR₉₅=672s [480-890s CI] | SLA overshoot: mean=234s, max=590s

H-09: Temporal Stability & Drift Detection BOTH*

“Is this agent getting worse over time during the test — or is it rock-solid from run 1 to run 30?”

H₀: Performance is stable — no systematic trend or structural change.
Hₐ: Performance is drifting — systematic trend or structural break exists.

Workflow:

  1. CUSUM: Track cumulative deviations from target (SLA or IQM baseline)
  2. EWMA: Weighted average emphasizing recent observations
  3. Change Point: Detect structural breaks (model updates)

Configuration Note: Metric selection for this hypothesis is configurable and can be changed per use case. Optional normalization may be applied when needed to align scales across metrics and fault categories.

Why: Static tests are a snapshot; CUSUM/EWMA detect trends before actual breach.

CUSUM S₃₀=8.4 (threshold h=15) | EWMA trend: +2.1s/run | Change point: None | STABLE

Power Caveat (n ≤ 30): CUSUM/EWMA designed for n>>100. At n=30, interpret drift verdicts as directional indicators, not definitive conclusions.

C.6 Key Concepts Glossary

ConceptOne-Sentence Explanation
Confidence Interval (CI)A range where we're 95% sure the true answer lies.
p-valueProbability that the observed difference is just random luck — below 5% = real.
Effect Size (A₁₂)How big a difference actually is — even if "significant," it might be tiny.
IQM25% trimmed mean — average of middle 50%, robust to outliers.
Bootstrap (BCa)Measuring uncertainty by reshuffling data 10,000 times. BCa corrects for bias and skewness.
Wilson CICI for success rates that works correctly with small samples or extreme values.
Kruskal-Wallis / Mann-WhitneyRank-based group comparisons — no bell-curve assumption needed.
Fisher's Exact TestExact probability test for success rates — works with small cell counts.
Levene's Test + CVVariance stability. CV = std/mean — shows reliability.
Wilcoxon Signed-RankNon-parametric test comparing observations against an SLA threshold.
Exact Binomial TestTests whether observed breach rate is within acceptable limit.
TOSTProves performance is within bounds, not just "not different from."
CVaRAverage of worst 5% of outcomes — measures tail severity.
Kaplan-MeierSurvival curve estimating probability of completing within a time limit.
CUSUMControl chart detecting cumulative drift from a target value.
EWMAWeighted moving average emphasizing recent observations for trend detection.
Error BudgetAllowed SLA breach count per window — tracks consumption toward violation.
Holm-BonferroniCorrection for multiple tests — prevents false positives from chance.
Worst-Category AssessmentJudge the agent by its worst fault category, not its best.
