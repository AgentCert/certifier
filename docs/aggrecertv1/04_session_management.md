# 04 — Session Management

## Responsibility

`main/services/cert_session_service.py` owns all reads and writes to the `certification_tasks`
MongoDB collection. Every other component interacts with cert task state exclusively through
this service. No other module accesses `certification_tasks` directly.

The existing `session_service.py` (for `pipeline_tasks`) is unchanged.

---

## Task Lifecycle — State Machine

```
                  ┌─────────┐
   POST received  │         │
  ─────────────►  │ PENDING  │
                  │         │
                  └────┬────┘
                       │  background worker starts
                       ▼
                  ┌─────────┐
                  │         │
                  │ RUNNING  │
                  │         │
                  └────┬────┘
                  /         \
       pipeline succeeds    any exception
              │                   │
              ▼                   ▼
        ┌──────────┐        ┌────────┐
        │ COMPLETED │        │ FAILED │
        └──────────┘        └────────┘
```

Terminal states are `COMPLETED` and `FAILED`. Once a task enters a terminal state its
document is never updated again.

Valid transitions:

| From | To | Trigger |
|---|---|---|
| — | PENDING | `create_task()` called by POST handler |
| PENDING | RUNNING | `set_started()` called by cert task runner |
| RUNNING | COMPLETED | `set_completed()` called by cert task runner on success |
| RUNNING | FAILED | `set_failed()` called by cert task runner on any exception |

`set_completed` and `set_failed` filter on `{"cert_task_id": ..., "status": "RUNNING"}` and
raise `ValueError` if no document matches (guards against double-write race).

`set_failed` is safe to call even if status is still `PENDING` (handles early failures before
`set_started` fires, e.g., semaphore acquisition failure at task start).

---

## Stages

| Stage name | Meaning |
|---|---|
| `pending` | Task created, background coroutine not yet started |
| `fetching_metrics` | Validating metrics_dir and counting matching documents |
| `running_pipeline` | Inside `run_pipeline()`: aggregation (Phase 2) + certification (Phase 3) |
| `storing_metadata` | Writing CertificationMetadata + AggregatedCategoryMetadata to MongoDB |
| `done` | Written by `set_completed()` on success |

> `storing_metadata` and `done` are observable but typically complete within 2–5 seconds.
> The GET polling endpoint will rarely capture these stages.


### Field Notes

- `result` is `null` until `set_completed()` writes it.
- `error` is `null` unless `set_failed()` writes it.
- `started_at` / `completed_at` are `null` until the relevant transition occurs.
- `storage_paths` uses paths relative to `certifier/` root for portability.
- `certification_id` is a new UUID generated by the cert task runner when the pipeline
  completes successfully. It links the `certification_tasks` document to the
  `certification_metadata` and `aggregated_category_metadata` documents.

## MongoDB Indexes

Created at startup in `main/main.py` (see `03_app_startup.md`):

```
certification_tasks:
  idx_cert_task_id_unique   {cert_task_id: 1}                unique
  idx_cert_agent_exp        {agent_id: 1, experiment_id: 1}
  idx_cert_status_created   {status: 1, created_at: -1}
  idx_cert_created_at       {created_at: 1}
```

The compound index on `(agent_id, experiment_id)` supports the duplicate-submission check:
`find_one({agent_id, experiment_id, status: {$in: ["PENDING","RUNNING"]}})`.

## Service Interface

```python
# main/services/cert_session_service.py

from motor.motor_asyncio import AsyncIOMotorCollection
from typing import Optional

class CertSessionService:
    def __init__(self, collection: AsyncIOMotorCollection):
        self._col = collection

    async def create_task(
        self,
        cert_task_id: str,
        agent_id: str,
        agent_name: str,
        experiment_id: str,
        certification_run_id: str,
        request_snapshot: dict,
    ) -> None:
        """
        Insert a new PENDING task document.
        Raises DuplicateKeyError on cert_task_id collision.
        """

    async def set_started(self, cert_task_id: str) -> None:
        """Transition PENDING → RUNNING. Sets started_at and stage='fetching_metrics'."""

    async def update_stage(self, cert_task_id: str, stage: str) -> None:
        """Update stage and updated_at. Called between pipeline steps."""

    async def set_completed(self, cert_task_id: str, result: dict) -> None:
        """
        Transition RUNNING → COMPLETED.
        Writes result dict, sets stage='done', sets completed_at.
        Raises ValueError if document is not in RUNNING state (guard against double-write).
        """

    async def set_failed(
        self,
        cert_task_id: str,
        error_code: str,
        message: str,
        failed_stage: str,
        detail: str,
    ) -> None:
        """
        Transition RUNNING → FAILED (or PENDING → FAILED for early errors).
        Writes error dict and completed_at.
        Safe to call even if status is PENDING.
        """

    async def get_task(self, cert_task_id: str) -> Optional[dict]:
        """Return the full task document or None if not found."""

    async def find_active_task(
        self, agent_id: str, experiment_id: str
    ) -> Optional[dict]:
        """
        Return a PENDING or RUNNING task for this (agent_id, experiment_id) pair.
        Used to reject duplicate submissions that would conflict on the workspace path.
        """
```

### All writes use `$set` + `$currentDate`

```python
# Example: set_started
await self._col.update_one(
    {"cert_task_id": cert_task_id, "status": "PENDING"},
    {
        "$set": {"status": "RUNNING", "stage": "fetching_metrics"},
        "$currentDate": {"started_at": True, "updated_at": True},
    },
)
```

Using `$currentDate` (server-side) instead of `datetime.utcnow()` avoids clock-skew issues
when multiple API instances run against the same MongoDB.

## Duplicate Submission Guard

Before creating a task the POST handler calls `find_active_task()`. If an active task exists
for the same `(agent_id, experiment_id)` pair, return HTTP 409 Conflict:

```json
{
  "status": "error",
  "error_code": "TASK_ALREADY_ACTIVE",
  "message": "A certification task is already RUNNING for agent_v2_4_1/exp_001",
  "details": {
    "cert_task_id": "existing-uuid",
    "status": "RUNNING",
    "stage": "running_pipeline"
  }
}
```

This guard prevents the workspace directory collision described in `00_overview.md`
(constraint 3): two concurrent tasks for the same `(agent_id, experiment_id)` would write
to the same `workspace/cert/{agent_id}/{experiment_id}/` directory.
