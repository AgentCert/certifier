2. Evaluation Methodology

  • Evaluation lifecycle: Each run evaluates the agent across four phases — detection, diagnosis, remediation, and post-remediation verification — within a controlled Kubernetes environment.
  • Numeric aggregation: Deterministic computation from telemetry — mean, median, P95, std dev, min, max for timing; success rates, false negative/positive rates, and token counts derived directly from run data.
  • Qualitative aggregation: LLM Council with k=3 independent judges per dimension + meta-reconciliation to produce consensus summaries, severity labels, and confidence ratings.
  • Inter-judge agreement: Computed as the proportion of evaluation dimensions where all 3 judges assigned the same severity label. Agreement of 1.0 means unanimous consensus across every dimension for every run. Currently 1.0 across all fault categories — all assessments rated Strong with High confidence.
  • Safety propagation: Worst-case policy — a single failure in any run flags the entire category; PII, hallucination, and malicious prompt detection tracked per-run with category-level rollup.
  • Metrics collected: Timing (TTD, TTM), success rates (detection, mitigation, FN, FP), quality scores (reasoning, response quality, action correctness), safety (hallucination, RAI, security, PII), and resource utilization (input/output tokens).
  • Fault taxonomy coverage: 3 of 5 canonical categories tested (Application, Network, Resource). Database/Storage and Security/Adversarial categories excluded from this certification run.
  • Aggregation pipeline: Per-run metrics (MongoDB metrics_runs) → fault-category scorecards → overall agent scorecard (aggregated_scorecards). Two-level rollup with run-count-weighted means for numerics and LLM Council consensus for narratives.
  • Statistical hypothesis testing: Beyond descriptive statistics, a 9-hypothesis inference framework (H-01 – H-09) is applied to produce confidence intervals, effect sizes, and worst-case guarantees.
  • Metrics not tracked: Recovery verification time (time from mitigation to confirmed healthy state) is not separately instrumented — it is included within TTM but not broken out. Future certification runs should capture this as an independent metric.
  • Run count note: The certification pipeline is configured for 30 runs per fault type. This certification run executed 5 runs per fault (15 total). Statistical measures such as P95 should be interpreted as directional with this sample size. The metadata field runs_per_fault: 30 reflects pipeline configuration, not actual execution count.

Statistical findings
Wilson intervals are preferred over normal-approximation intervals because they remain well-calibrated at the boundaries (0% and 100%) where this agent's proportions cluster. Two key signals emerge:
  • Detection is uncertifiable. The figure is reported per category, not pooled: the best per-category Wilson lower bound is only 9.5% (Application and Resource, each 6/30), and Network's per-category floor is 0% with an upper bound of 11.4% — below the 60% minimum-rate threshold. A larger sample cannot rescue this dimension without a real capability improvement.
  • Mitigation is certifiable. With 30 successes out of 30 per category, the Wilson lower bound 88.6% already exceeds the 85% mitigation target — publishable as a per-category SLA.
Method: Wilson 95% CI on proportions.
Median & P95. The median is robust to outliers — a single anomalously slow run does not distort the typical performance. P95 (95th percentile) captures tail latency: "95% of runs complete within this time," giving a practical upper bound for SLA evaluation. At the current per-category sample size (n = 30), P95 should be read as a directional indicator of tail behaviour, not a statistically certified bound.

Detection vs. mitigation. Detection rate measures whether the agent explicitly identified the fault before acting; mitigation rate measures whether the fault was resolved by end-of-run, regardless of detection. The agent may remediate a fault through its standard operational workflow (e.g., restarting unhealthy pods, cleaning disk) without classifying it as a "detected fault." That is why mitigation can reach 100% while detection stays low — the agent acts on symptoms even when it does not formally flag a root cause.
