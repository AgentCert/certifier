2. Key Definitions

2.1 Detection vs. Mitigation

Detection rate measures whether the agent explicitly identified the fault before acting. Mitigation rate measures whether the fault was resolved by the end of the run, regardless of detection. The agent may remediate a fault through its standard operational workflow (e.g., restarting unhealthy pods, cleaning disk) without classifying it as a "detected fault." This explains why mitigation can reach 100% while detection remains low — the agent acts on symptoms even when it does not formally flag a root cause.
Metric Definition Scope
Detection Rate Agent explicitly identifies the injected fault as the root cause Per-run binary (detected / not detected)
Mitigation Rate Fault is resolved by end of run, regardless of detection method Per-run binary (mitigated / not mitigated)
Time-to-Detect (TTD) Elapsed seconds from fault injection to first detection signal Only for runs where detection = true
Time-to-Mitigate (TTM) Elapsed seconds from fault injection to confirmed remediation Includes detection time; not time from detection to fix
False Positive Rate Agent flags a fault that was not injected Per-run count, aggregated to rate
False Negative Rate Agent fails to detect an injected fault 1 − Detection Rate

2.2 Why Median and P95?

Median is robust to outliers — a single anomalously slow run does not distort the typical performance. P95 (95th percentile) captures tail latency: "95% of runs complete within this time," providing a practical upper-bound for SLA evaluation.

Statistical note: With small sample sizes (e.g., n=5 runs per category), P95 equals the maximum observed value. These percentiles should be interpreted as directional indicators, not statistically robust bounds.

3. Wilson Score Confidence Intervals

Wilson intervals are preferred over normal-approximation intervals because they remain well-calibrated at the boundaries (0% and 100%) where this agent's proportions cluster. They provide reliable confidence bounds even with moderate sample sizes and extreme success/failure rates.

3.1 Formula

Wilson CI = (p̂ + z²/2n ± z · √(p̂(1−p̂)/n + z²/4n²)) / (1 + z²/n)

where:
  p̂ = observed proportion (successes / trials)
  n = number of trials
  z = z-score for desired confidence level (1.96 for 95%)

4. Certification Signals

Three key signals emerge from the statistical evaluation. These determine whether a dimension is certifiable, needs improvement, or is already strong enough to publish as an SLA.
Detection is uncertifiable. Even the best-performing categories (Application, Resource) have a certified floor of only 9.5%. The network upper bound 11.4% is below the 60% minimum-rate threshold, so a larger sample cannot rescue this dimension without a real capability improvement.
Mitigation is certifiable. With 30 successes out of 30 per category, the Wilson lower bound 88.6% already exceeds the 85% mitigation target. Pooled across all 90 runs, the floor rises to 95.9% — publishable as an SLA.
Zero false positives. The 0/90 false-positive count has a one-sided Wilson upper bound near 4.1%, which is well inside noise tolerance.

6. Sample Size & Confidence Guidance

The number of runs directly impacts statistical confidence. Below is a reference for how Wilson CI width narrows as sample size increases, assuming a 100% observed success rate.
Runs per Fault Observed Rate Wilson Lower Bound (95%) CI Width Hypothesis Testing
5 100.0% 56.6% 43.4% Skipped — insufficient sample
10 100.0% 72.2% 27.8% Skipped — insufficient sample
20 100.0% 83.9% 16.1% Marginal — below 85% floor
30 100.0% 88.6% 11.4% Valid — exceeds 85% threshold
50 100.0% 92.7% 7.3% Strong — high confidence
Why 30 runs? At n=30 with a perfect success rate, the Wilson lower bound (88.6%) first crosses the 85% SLA certification threshold. Below 30 runs, even 100% observed success cannot guarantee the SLA floor with 95% confidence — hypothesis testing is skipped and results are reported as descriptive only.
