The Big Picture — Where Do These Modules Fit?

Upstream

Metrics Extractor

Produces per-run metric documents stored in MongoDB agent_run_metrics collection. Each doc contains quantitative + qualitative fields from a single agent run.

Aggregator

Multi-Run Aggregation

Takes N per-run documents, computes statistical aggregates, derived rates, boolean rollups, and LLM-synthesized textual consensus. Produces a CertificationScorecard.

Cert Builder

Report Generation

Transforms the aggregated scorecard into a structured 12-section certification report with tables, charts, LLM narratives, and findings.

Aggregator — Orchestration Flow

Step 1: Query

Fetch per-run metric documents from MongoDB (MetricsQueryService) or local JSON files (DirectoryQueryService). Each doc = one agent run with quantitative + qualitative fields.

Step 2: Compute
Path A
Numeric Aggregation

mean, median, std_dev, p95, min, max for timing, quality, tokens, counts

Path B
Derived Rates

detection, mitigation, false neg/pos, RAI compliance, security compliance

Path C
Boolean Aggregates

PII detection (any_detected, rate) + hallucination detection

Path D
Textual Synthesis

LLM Council: k judges + meta-judge for narrative consensus

Step 3: Assemble

ScorecardAssembler merges all 4 paths into a FaultCategoryScorecard, repeats for each category, then combines into the final CertificationScorecard.

Step 4: Store

ScorecardStorage upserts to MongoDB aggregated_scorecards collection (keyed by certification_run_id or agent_id) + outputs JSON file.

Aggregator — LLM Council Pattern

Architecture

k-Judge + Meta-Reconciliation

Judge 1

Judge 2
temp=0.3
Judge 3
temp=0.3
↓ ↓ ↓
Meta-Judge (temp=0.1) — Reconciles k outputs
↓
TextualConsensus: summary + severity + confidence + agreement
Textual Metrics Synthesized
  1. rai_check_summary — RAI compliance narrative
  2. overall_response_and_reasoning_quality — Quality analysis
  3. security_compliance_summary — Security posture
  4. agent_summary — Overall agent assessment
  5. known_limitations — Ranked by severity & frequency
  6. recommendations — Prioritized action items
Error Handling

If a judge fails, it returns a default:

{ "consensus_summary": "Judge evaluation failed.", "severity_label": "Weak", "confidence": "Low" }
Config
{ "council_size": 3, "judge_temperature": 0.3, "judge_max_tokens": 1500, "meta_judge_temperature": 0.1, "meta_judge_max_tokens": 2000 }
Key Detail

Categories processed sequentially (not parallel) to manage LLM API rate limits.

CertificationPipeline in certification_pipeline.py manages the 4 phases sequentially. Intermediate JSON files are written to a temp directory. Debug mode copies them to data/intermediate/.

Cert Builder — Phase 2: Computation (6 Builders)

Builder A

Scorecard

7 normalized dimensions (0–1 scale) + severity-tagged findings

Normalization:

  • Speed: 1 - mean/1800
  • Scores: score / 10
  • Hallucination: inverted
  • Rates: already 0–1, clamped
Builder B

Tables

13 structured data tables

TTD stats, TTM stats, detection rates, safety summary, action correctness, reasoning quality, hallucination, RAI compliance, security compliance, token usage, limitations, recommendations, judge models

Builder C

Charts

9 chart data structures

Radar (scorecard), TTD bar, TTM bar, rates bar, accuracy heatmap, reasoning bar, hallucination bar, compliance bar, token stacked bar

Depends on Scorecard builder

Builder D

Assessments

Per-category LLM assessment passthrough

agent_summary, response_reasoning_quality, security_compliance, rai_compliance — reformatted from Phase 1 textual metrics

Builder E

Hardcoded

Static content from YAML

Metric definitions, normalization formulas, methodology bullets, section introductions — no computation needed

Builder F

Cards

3 identity/scope cards

Identity card (agent name, ID), scope card (run counts, dates), categories card (fault types tested)

Critical Contract Points
  • Field names must match exactly (e.g., time_to_detect)
  • Stats must include mean, median at minimum
  • Derived rates are expected as 0–1 floats
  • Textual metrics contain consensus_summary, severity_label
  • Categories must have faults_tested array

Key Patterns & Takeaways

Architecture decisions, testing, and debugging

Architecture Patterns

Pattern 1

LLM Council (Aggregator)

  • k independent judges for diverse perspectives
  • Meta-judge reconciles divergent opinions
  • Lower temperature on meta-judge (0.1) for stability
  • Graceful fallbacks on judge failure
Pattern 2

Pluggable Query Service

  • MetricsQueryService — MongoDB for production
  • DirectoryQueryService — Local files for dev/test
  • Same interface, swappable backends
Pattern 3

Concurrent + Sequential LLM

  • Aggregator: categories processed sequentially (rate limits)
  • Cert Builder Phase 3: calls 1–5 concurrent, call 6 sequential (dependency)
Pattern 4

Phase-Based Pipeline

  • Each phase writes intermediate JSON
  • Debug mode preserves intermediates
  • Phases are independently testable
  • Pydantic validation at each boundary
Pattern 5

Builder Dependency Graph

  • Phase 2 uses explicit dependency ordering
  • Scorecard runs first (independent)
  • Charts wait for scorecard dimensions
  • All results merged and validated as ComputedContent
Pattern 6

Lazy Loading & Config

  • Module __init__.py uses lazy imports
  • Config loaded once, cached globally
  • YAML prompts & configs separated from code

Testing & Debugging

Aggregator Tests

685 lines of tests

  • Schema tests — All 12 Pydantic models
  • Numeric tests — compute_stats, aggregates, derived rates, boolean rollups
  • Config/prompt tests — File existence + valid format
  • LLM Council tests — Mocked judge/meta-judge, error handling
  • Assembly tests — Scorecard construction
Demo Notebook

aggregator/notebooks/aggregation_demo.ipynb provides interactive walkthrough with sample data.

Debugging Tips
  • Run cert_builder with debug=True to save intermediate JSONs
  • Check data/intermediate/ for phase-by-phase outputs
  • Aggregator outputs aggregated_scorecard_output.json — inspect before cert_builder
  • LLM failures default to fallback templates — check for "evaluation failed" strings
  • Token usage tracked across all LLM calls — check for runaway costs
Common Issues
  • Missing Azure env vars — Phase 3 will fail silently with fallbacks
  • Schema mismatch — Aggregator field names must match cert_builder expectations
  • Rate limits — Sequential category processing in aggregator helps
  • Empty metrics — Both modules handle None/missing gracefully
