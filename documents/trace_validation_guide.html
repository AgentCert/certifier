1. Expected Trace Structure

A valid AgentCert experiment trace contains these spans in the following order:

experiment-run  (name: experiment-run, depth 0)  →  top-level span with experiment metadata, resiliency score, fault counts
    experiment-triggered  (depth 1, child)  →  marks the experiment trigger event
    agent_onboarding  (depth 0 or 1)  →  agent metadata: agent_id, agent_name, agent_version, capabilities, tools
    fault: <name>  (name: fault: *, depth 1) × N  →  one per injected fault, carries ground truth and fault attributes

… remaining spans (agent scans, LLM calls, MCP tool calls) in any order …

How fault injection spans are identified

Fault injection spans are identified by the name field matching the pattern fault: * (e.g. fault: pod-delete, fault: pod-network-loss, fault: disk-fill). The span type is SPAN. The fault bucketing pipeline uses this naming convention to deterministically create per-fault buckets.

Typical agent scan spans (after fault injections)

These can appear in any order after the setup spans. Common span names include:

tool_selectionGENERATIONLLM picks the data source (e.g. kubernetes or prometheus)
Received Proxy Server RequestGENERATIONLiteLLM proxy forwarding the LLM call

2. Anchor Presence (IDs & Metadata)

Every span in the trace should carry identifying anchors so events can be correlated back to the correct experiment run.

Check 2.1 — experiment_id & run_id on every span
  • Both experiment_id (or experiment.id) and run_id (or experiment.run_id / experiment_run_id) should ideally be present in every span's metadata or attributes.
  • Verify these values are consistent across all spans in the trace — a single experiment run produces one unique pair.
  • On experiment-run and fault: * spans, look in metadata.attributes.
  • On GENERATION spans (agent scans), look in metadata.requester_metadata.
  • If any span is missing these IDs, it cannot be reliably attributed to the experiment and downstream bucketing may fail.
Check 2.2 — agent_id in onboarding span
  • The agent_onboarding span must contain agent_id, agent_name, and agent_version in its input or metadata.
  • The experiment-run span should also carry agent_name and agentid in its attributes.
  • These values propagate into every FaultBucket created by the bucketing pipeline.

3. Context Leakage to Agent

The agent under test must not have visibility into the experiment setup, fault configurations, ground truth, or injection timing. Any such leakage invalidates the experiment.

Check 3.1 — Fault injection span data is invisible to the agent
  • The fault: * spans carry ground truth, fault attributes (fault.target_label, fault.verdict, fault.probe_success_pct, etc.), ideal course of action, and SLA definitions.
  • None of this data must appear in the input to any agent-facing LLM call (the input field of tool_selection or llm_analysis GENERATION spans).
  • The agent should only see symptoms (e.g. pod status, events, logs) — never the fault definition or experiment design.
Check 3.2 — Ground truth not leaked in LLM prompts
  • Search all GENERATION span inputs for keywords: ground_truth, ideal_course_of_action, ideal_tool_usage_trajectory, remediation, detection_signals, symptoms.
  • If any of these appear in agent-facing prompts, the experiment is compromised.
Check 3.3 — Experiment metadata not in agent prompts
  • The experiment-run span carries fields like experiment.total_faults, experiment.faults_passed, experiment.resiliency_score, and the fault list from onboarding.
  • These must not appear in the agent's LLM input — the agent should not know how many faults were injected, their names, or expected outcomes ahead of time.
Check 3.4 — No temporal hints about injection windows
  • The agent should not receive fault injection timestamps (fault.started_at, fault.finished_at) or span start/end times from the fault: * spans.
  • Verify no agent-facing data passes injection timing information.

4. MCP Information Leakage

The MCP tool is the agent's interface to the cluster. It must only return observable operational data — not experiment internals.

Check 4.1 — MCP does not return experiment orchestration details
  • Inspect the data returned by MCP tool calls (visible in the input field of the subsequent llm_analysis GENERATION span).
  • MCP responses should not include fault parameters such as target labels, chaos duration, probe definitions, or fault verdict from the experiment framework.
Check 4.2 — MCP does not expose ground truth or expected outcomes
  • Verify the MCP data does not contain expected probe success criteria, remediation steps, or pre-configured pass/fail thresholds that would reveal the experiment design.
Check 4.3 — MCP does not leak fault injection pod/container activity
  • Chaos runner pods, helper pods, and experiment executor containers should be filtered from the data returned to the agent, or at minimum not confused with application workloads.

5. MCP Data Relevance

While MCP should not leak experiment internals, it must provide sufficient operational data for the agent to detect faults through symptoms.

Check 5.1 — MCP returns adequate data for analysis
  • Inspect the data passed to the llm_analysis prompt. It should include relevant operational signals: pod statuses, restart counts, event summaries, and resource states.
  • Verify that during active fault injection, the MCP data reflects the actual degraded state of affected pods/services.
  • If the data consistently shows "everything healthy" while faults are active, the MCP is not surfacing the right information.
Check 5.2 — Key data fields are populated
  • Check for presence and non-emptiness of key MCP data keys (visible in requester_metadata.mcp_data_keys): pod listings, events, resource metrics.
  • Verify pods_by_status accurately reflects the cluster state at scan time.
  • Verify events_warning count is non-zero during fault windows when warnings are expected.
Check 5.3 — MCP errors are absent or justified
  • Check the mcp_errors field in requester_metadata. It should be null or empty for successful scans.
  • Persistent MCP errors indicate the agent is not getting any useful data for analysis.

6. Span Completeness & Ordering

Check 6.1 — All expected span types are present
  • Verify the trace contains:
    1. One experiment-run span
    2. One agent_onboarding span
    3. The expected number of fault: * spans (should match experiment.total_faults from the experiment-run span)
Check 6.2 — Correct ordering of setup spans
  • experiment-run must come first (earliest startTime among depth-0 spans).
  • agent_onboarding must come after experiment-run and before fault: * spans.
  • fault: * spans must come after agent_onboarding.
  • Spans after the fault injections (agent scans, LLM calls) can appear in any order.
Check 6.3 — Valid timestamps
  • All startTime values must be valid ISO-8601 timestamps.
  • endTime ≥ startTime for every span.

7. Ground Truth Integrity

Check 7.1 — Every fault: * span carries ground truth
  • Each fault: * span must have a ground_truth object in its metadata containing: fault_description_goal_remediation (with symptoms, goal, remediation), ideal_course_of_action, ideal_tool_usage_trajectory, and sla.
  • Missing ground truth means the downstream metrics extraction pipeline has nothing to evaluate against.
Check 7.2 — Fault attributes are complete
  • Each fault: * span should have these attributes populated in metadata.attributes: fault.name, fault.target_namespace, fault.target_label, fault.target_kind, fault.verdict, fault.probe_success_pct, fault.status, fault.started_at, fault.finished_at.
Check 7.3 — Fault name consistency
  • The fault name extracted from the span name (fault: pod-delete → pod-delete) must match the fault.name attribute inside the span's metadata.

8. Agent Issue Detection Ability

Evaluate whether the agent's LLM analysis correctly identifies issues from the data it receives.

Check 8.1 — Agent detects application-level impact
  • For each injected fault, check whether the agent's llm_analysis output identifies the affected service or pod — not just engine names or generic status.
  • The agent should correlate symptoms (pod status, restart count, error events) to specific application degradation.
  • If every fault shows "insufficient data" for impact, the agent is failing at correlation.
Check 8.2 — Agent correctly identifies fault targets
  • The target_app field in the agent's structured analysis output should match the actual fault target.
  • A null target across all faults indicates the agent cannot determine what was affected.
Check 8.3 — Health score reflects actual system state
  • Compare the agent's reported overall_health_score against the known fault state at that time.
  • A high health score during a period when all probes report 0% success is a major discrepancy.
Check 8.4 — Agent detects fault recovery
  • After a fault window ends, verify whether the agent recognizes the system returning to normal.
  • Persistent "insufficient data" in recovery fields indicates no recovery detection.
