01

What is AgentCert?

Understanding the problem, purpose, and expected business outcomes.

The Problem We're Solving

Why AgentCert exists and what gap it fills.

The Challenge

  • Autonomous IT-Ops agents are making real-time decisions in production environments
  • No standardized way to evaluate whether these agents are reliable, safe, and effective
  • Agents may hallucinate, take collateral actions, miss faults, or violate safety policies
  • Stakeholders need confidence before deploying agents in critical infrastructure

The Solution: AgentCert

  • A rigorous certification framework for evaluating IT-Ops agents
  • Uses simulated fault-injection experiments with known ground truth
  • Extracts quantitative + qualitative metrics per fault run
  • Aggregates across multiple runs for statistical confidence
  • Generates a professional certification report with a verdict

Business Outcome: A certified agent means the organization has evidence-backed confidence that the agent detects faults reliably, mitigates them safely, and complies with responsible AI and security standards.

What AgentCert Evaluates

Five certification categories that determine the agent's verdict.

Detection & Response

  • Time to Detect (TTD)
  • Detection Accuracy
  • Mean Time to Mitigate
  • Resolution Success Rate
  • Number of Steps

Accuracy & Efficiency

  • Root Cause Accuracy
  • Tool Selection Accuracy
  • Token Usage & Costs
  • API Calls Efficiency
  • Execution Time

Reasoning & Quality

  • Reasoning Quality Score
  • Plan Adherence
  • Hallucination Score
  • Exploration vs Exploitation
  • Confidence Calibration

Safety & Compliance

  • PII Detection
  • RAI Compliance
  • Security Compliance Status
  • Malicious Prompt Detection
  • Collateral Damage Assessment

Resource Utilization

  • Token Usage (Input/Output)
  • LLM Cost Tracking
  • Rollback Actions
  • Content Safety
  • Sustainability Metrics

End-to-End Pipeline

How data flows from raw Langfuse trace to final certification report.

The AgentCert Pipeline

Four major stages transform raw agent traces into a certification verdict.

Phase 1a
Fault Bucketing
Split multi-fault trace into per-fault buckets
→
Phase 1b
Metrics Extraction
Extract quant + qual metrics per bucket
→
Phase 2
Aggregation
Aggregate ~30 runs into scorecards
→
Phase 3
Certification
Generate report with verdict

Pipeline Runner 1

run_bucketing_and_extraction_pipeline.py

  • Takes a raw Langfuse trace JSON
  • Runs fault bucketing (LLM classification)
  • Runs metrics extraction per bucket
  • Outputs per-run *_metrics.json files

Pipeline Runner 2

run_aggregation_and_certification_pipeline.py

  • Takes a directory of *_metrics.json files
  • Runs aggregation (numeric + LLM Council)
  • Runs certification report builder
  • Outputs final certification report (JSON/HTML/PDF)

Data Flow: Trace to Certification

Each stage produces specific artifacts that feed the next stage.

Raw Langfuse Trace (JSON) | v [Phase 1a] Fault Bucketing # fault_analyzer/ | Splits multi-fault trace into per-fault event buckets | Output: per-fault bucket JSONs + manifest.json v [Phase 1b] Metrics Extraction # metrics_extractor/ | LLM extracts per-batch metrics, code aggregates numerics | Output: per-run *_metrics.json (quant + qual + tokens) v [Phase 2] Aggregation # aggregator/ | ~30 runs per fault -> statistical aggregation | Numeric: code (mean, median, p95, std_dev) | Textual: LLM Council (k judges + meta-judge) | Output: aggregated_scorecard.json v [Phase 3] Certification Builder # cert_builder/ | Ingestion -> Computation -> Narratives -> Assembly | Output: certification_report.json (12-section report) v Final Verdict: Certified / Not Certified

04

Module Responsibilities

What each major module does and the artifacts it produces.

Module: fault_analyzer/

Phase 1a — Splits multi-fault Langfuse traces into per-fault event buckets.

What It Does

  1. Loads raw Langfuse trace, sorts events chronologically
  2. Extracts FAULT_DATA (injected ground truth)
  3. Batches events (default batch size: 10)
  4. Sends batches to LLM for fault classification
  5. Creates per-fault buckets with lifecycle phases (detection, investigation, remediation, verification)
  6. Enriches buckets with ground truth metadata
  7. Writes per-fault JSON + manifest

Key Components

ComponentPurpose
FaultBucketingPipelineMain orchestrator class
FaultEventClassifierLLM-based batch classifier
FaultBucketContainer for fault lifecycle events
fault_bucketing_config.jsonModel, temperature, batch settings
prompt.ymlLLM system prompt template

Input: Raw Langfuse trace JSON (interleaved multi-fault events)  →  Output: Per-fault bucket JSONs + manifest.json

Module: metrics_extractor/

Phase 1b — Extracts quantitative and qualitative metrics from each fault bucket.

Extraction Process

  1. Takes per-fault bucket (from Phase 1a)
  2. Splits spans into batches of 15
  3. LLM extracts metrics per batch
  4. Code-based aggregation for all numeric values (LLM never does math)
  5. Identifies detection/mitigation timestamps from spans
  6. Tracks token usage
  7. Optionally stores to MongoDB

Output Schema: ExtractionResult

Quantitative

  • TTD, TTM, fault detection status
  • Trajectory steps, tokens, tool calls
  • PII detection, malicious prompts detected
  • Tool selection accuracy

Qualitative

  • RAI check, security compliance
  • Reasoning quality, hallucination score
  • Agent summary, plan adherence, collateral damage

Input: Per-fault bucket JSON  →  Output: *_metrics.json (per-run metrics document)

Module: aggregator/

Phase 2 — Aggregates ~30 per-run metrics into fault-category and agent-level scorecards.

Aggregation Strategy

Numeric (Code)

  • Mean, median, std_dev
  • P95, min, max, sum
  • Detection/mitigation rates
  • False positive/negative rates

Textual (LLM Council)

  • k independent judges
  • Meta-judge reconciliation
  • Consensus summary
  • Ranked limitations & recs

Key Components

ComponentPurpose
AggregationOrchestratorMain pipeline orchestrator
LLMCouncilMulti-judge consensus system
numeric_aggregation.pyPure numeric stat functions
compute_stats()Statistical aggregation helper
compute_derived_rates()Detection/mitigation rate calc

Safety Rule: Worst-case propagation for safety metrics — if any single run fails safety checks, the entire fault category is marked as failed.

Key Design Principles

Fundamental architectural decisions that shape how every module works.

Design Principles

These principles are non-negotiable and inform every code decision.

LLM Never Does Math

All numeric operations (sums, ratios, averages, percentiles, rates) are executed in deterministic Python code. The LLM extracts values; code aggregates them. This eliminates hallucinated calculations.

Hybrid Aggregation

LLM extracts per-batch qualitative values. Code aggregates all numerics. Textual synthesis uses an LLM Council (k judges + meta-judge) to reduce single-model bias.

Worst-Case Safety Propagation

If any single run fails a safety check (RAI, PII, security), the entire fault category is marked as failed. Safety is never averaged — it's a strict boolean gate.

Statistical Rigor (~30 Runs Per Fault)

Each fault is repeated ~30 times to capture variance. This gives enough samples for meaningful means, percentiles, and confidence intervals rather than anecdotal single-run results.

Ground Truth Comparison

Fault configuration provides ground truth that the agent never sees. This allows objective measurement of detection accuracy, root cause identification, and mitigation correctness.

Modular Pipeline, Explicit Interfaces

No shared state or global singletons. Each phase receives explicit inputs and returns explicit outputs. Builder scripts are plain functions. Only assemblers and the pipeline orchestrator use classes.
